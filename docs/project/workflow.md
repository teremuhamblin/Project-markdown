# Workflow du projet

Ce document décrit le workflow officiel du projet :  
comment travailler, contribuer, organiser les tâches et maintenir la cohérence du dépôt.

---

## 1. Objectifs du workflow
- Standardiser la manière de travailler sur le projet.
- Faciliter la contribution et la relecture.
- Garantir une qualité constante dans tous les fichiers.
- Réduire les erreurs, doublons et incohérences.

---

## 2. Processus de travail

### Étape 1 — Comprendre le contexte
Avant toute modification :
- Lire le `README.md`
- Vérifier les règles dans `writing/rules.md`
- Consulter la structure dans `project/structure.md`

Objectif : éviter les erreurs de style ou de structure.

---

### Étape 2 — Créer ou modifier un fichier
- Respecter les règles d’écriture.
- Suivre la structure standard d’un fichier Markdown.
- Ajouter des exemples si nécessaire dans `exemples.md`.

---

### Étape 3 — Vérification
Avant de valider :
- Relire le fichier.
- Vérifier la cohérence avec les autres documents.
- S’assurer que le ton est clair, professionnel et minimaliste.

---

### Étape 4 — Commit
Règles pour les messages de commit :
- Format : `type: description courte`
- Exemples :
  - `add: ajout du fichier workflow.md`
  - `update: amélioration des règles d’écriture`
  - `fix: correction d’une faute`

---

### Étape 5 — Mise à jour de la documentation
Si une règle, structure ou convention change :
- Mettre à jour le fichier concerné.
- Ajouter une entrée dans `CHANGELOG.md`.

---

## 3. Workflow de contribution

### Pour ajouter un nouveau fichier
1. Vérifier qu’il n’existe pas déjà.
2. Créer le fichier dans le bon dossier.
3. Ajouter un titre clair.
4. Documenter son objectif.
5. Ajouter un exemple si pertinent.

---

### Pour modifier un fichier existant
1. Lire entièrement le fichier.
2. Identifier les incohérences ou améliorations.
3. Modifier en respectant les règles.
4. Documenter les changements dans le commit.

---

### Pour ajouter un nouveau dossier
1. Justifier son existence.
2. Créer un `README.md` dans le dossier.
3. Mettre à jour `structure.md`.

---

## 4. Bonnes pratiques
- Toujours privilégier la clarté.
- Ne jamais laisser de texte ambigu ou incomplet.
- Garder les fichiers courts et bien structurés.
- Documenter chaque changement.
- Maintenir une cohérence globale dans tout le projet.

---

## 5. Maintenance continue
- Revoir régulièrement les règles.
- Simplifier quand c’est possible.
- Garder la documentation à jour.
- Supprimer les fichiers obsolètes.

---

## 6. Objectif final
Un projet **propre**, **cohérent**, **professionnel**, facile à lire, à maintenir et à faire évoluer.
